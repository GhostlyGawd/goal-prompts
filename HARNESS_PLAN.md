# HARNESS_PLAN — goal-prompts as the decision layer of a build harness

2026-07-09 · Fable 5 session output. North star: systems that generate
revenue while operated by Opus alone. This plan records the architecture,
what was built in this session (most of the load-bearing layer), what Opus
builds next inside the rails, and what stays deferred. Every load-bearing
choice has an ADR in DECISIONS.md; the specs live in specs/.

## 1. The one-paragraph theory

Opus fails predictably: it rationalizes past instructions, claims green
without running, improvises beyond spec, and drifts scope. It does not
rationalize past a failing exit code. So the harness converts every
trusted judgment into a command that exits non-zero — and reserves frontier
capability (this session) for building those commands, the one job Opus
must never do for itself. Goal-prompts already worked this way for its own
content (`build.py` lints every brief; `scripts/check` is the whole gate;
CI fails on drift), which is why it is the right decision layer: the
conventions the harness needs were already load-tested here.

## 2. Architecture — three layers

```
DECIDE   goal-prompts catalog (this repo) — fetched into the USER'S ACTUAL
         REPO; every stage below runs there, no side workspaces (ADR-11)
         Founder Funnel 61→67: is it worth building?   → VERDICT.md
         Build family  141→144: will it ship?          → the repo on rails
         Meta/Act 29, 46→47: is it healthy? what next? → the standing loop
              │  briefs are fetched read-only (raw/<id>.md, MCP, plugin)
              ▼
ENFORCE  template/ — the golden-path harness 141 installs into that repo,
         greenfield or grafted onto existing code (built + verified this session)
         scripts/check = spec lint → tests → evals, one exit code
         four enforcement moments, all deterministic:
           after every edit   (.claude PostToolUse → exit 2 on red)
           before every commit (.githooks/pre-commit)
           on every push       (.github/workflows/check.yml + branch protection)
           at the ship gate    (brief 144 re-runs everything adversarially)
         plus a fifth that blocks rather than checks:
           harness-layer edits (.claude PreToolUse → exit 2, operator-owned paths)
              ▼
OPERATE  the repo ships; 29 · Weekly Vitals trends it;
         46 · Triage routes findings; 47 · The Fixer lands them as commits
```

**Opus-buildable inside the rails:**

- Product repos: everything downstream of `141` — spec content, product code,
  product tests, eval cases. The rails were built precisely so this is safe.
- Example artifacts for the Build briefs (`example:` front matter pointing at
  a real SCAFFOLD.md/BUILDLOG.md once the first product exists) — additive,
  linted by build.py:144.
- Routine catalog maintenance under `scripts/check` (typos, related: links).

**Deferred (with reasons):**

- Second-stack template (Node/web): defer until a verdict demands it —
  porting the enforcement moments needs frontier verification (queue item).
- Catalog monetization (sponsored/collab slots): zero-audience today
  (ADR-9); revisit when raw/ fetch counts or stars move.
- Headless automation of the weekly loop (cron → Claude headless running 29):
  cheap, but pointless before a product exists to operate.
- Any orchestrator/dashboard: ruled out (ADR-7).

## 5. Opus failure modes → the gate that catches each

| Failure mode | Deterministic catch |
|---|---|
| Claims tests pass without running them | PostToolUse hook runs the gate after every edit; exit 2 blocks |
| Commits on red / stacks work on a broken state | .githooks/pre-commit; CI required status check |
| Edits the gate to make red green | PreToolUse hook blocks harness paths outright; CODEOWNERS routes stray diffs to the operator; 144 diffs harness vs template |
| Vague, untestable acceptance criteria | spec_lint AC grammar: no `check:` command → parse error |
| Improvises beyond spec | only-source-of-work rule + 143's scope gate; unspecced work has no AC and no pinning test to point at |
| Quiet dependency creep | spec_lint: package in requirements.txt with no ADR naming it → red |
| Claims AC built without proof | spec_lint: `status: built` with no `ac_<n>` test → red |
| Weakens a product test to get green | **not hook-catchable** (Opus owns product tests). Mitigated: 144's sabotage lens, CLAUDE.md prohibition, operator diff review. Named residual risk (ADR-8) |
| Builds taste-only products the system can't verify | 141 Phase 2 lens 2 refuses; spec grammar can't express an uncheckable AC (ADR-6) |

## 6. Sequencing — what happens next, in order

1. **Operator** (one-time, ~15 min): merge this branch; pick the repo — the
   existing repo the product lives in, or a fresh one for a new idea — and
   the idea question (see Open Questions).
2. **Opus, in that actual repo**: run the master conductor
   (`raw/playbook-zerotoship.md`) — 61→67 write the venture reports at the
   repo root, `67` rules go/pivot/kill at its ask-first gate (operator
   accepts or stops), then 141 installs the harness (greenfield or graft;
   operator approves the plan and does the GitHub duties when asked),
   142 writes the spec (operator ratifies), 143 implements AC by AC,
   144 rules ship or hold. A repo with the decision already made starts at
   `raw/playbook-buildloop.md` instead.
3. **Opus, operate layer**: same repo — weekly `29`; `46` → `47` on
   accumulating findings; `144` re-run before each meaningful release.
4. **Fable, next session**: work FABLE_BUILD_QUEUE.md top-down — first item
   is supervising/reviewing the first real pass through steps 2–3, because
   the first traversal is the eval of the harness itself.

## 7. Open Questions (operator-only; nothing here blocks the build)

1. **Niche/domain input for the Founder Funnel** — 61–67 need a domain to
   mine. What fields does the operator know well enough to judge demand
   evidence? (The harness verifies mechanics, not market taste.)
2. **Budget** — monthly API spend ceiling for Opus sessions, and any cap on
   GitHub Actions minutes; determines weekly-loop cadence.
3. **Payment rails** — 144's "first dollar" lens needs a human-owned
   account (Stripe/Gumroad/Lemon Squeezy?). Which one exists or is easiest?
4. **GitHub handle for CODEOWNERS** and whether the operator will enable
   branch protection on product repos (the one merge-gate step no agent can
   perform).
5. **Appetite** — start with the full Founder Funnel, or does the operator
   already have a product decision (in which case step 2 collapses to
   writing a one-paragraph verdict by hand and starting at 141)?

## 8. What "done" looks like

One product repo where: every AC in SPEC.md is `built` and test-pinned, CI
is a required check that has blocked at least one red push, SHIP-GATE.md
says ship with a first-dollar path that was actually walked, HEALTH.md has
three weekly rows of trend arrows — and no Fable session was needed after
the scaffold. That repo is the harness's own eval case.
