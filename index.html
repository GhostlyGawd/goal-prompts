01The problem

“Review my repo” gets a different answer every time.

Point an agent at your own code and you get a wall of vibes one run, a shrug the next. The fix isn’t a bigger model — it’s structure: a brief that tells the agent exactly what to look for, how to weigh it, and what to leave behind.

Every brief here is tested on this repo before it ships. The reports are public, and the fix log traces each shipped fix back to the finding that surfaced it.

02How it works

One method. Every audit.

Whatever you’re checking — bugs, speed, security, design, your AI agent itself — every brief walks the same arc. That’s what makes the output consistent and repeatable.

1

Explore

The agent maps the repo and finds where the risk actually lives.

2

Audit

It sweeps through a fixed set of expert lenses, citing file & line.

3

Curate

It keeps only what it can prove, ranked by real user impact.

4

Report

It writes one structured file at your repo root — the deliverable.

→

Act

Turn findings into commits with the Studio & the Fixer brief.

Read-only by default

Every audit brief reads your code and runs read-only checks. Its only write is its own report.

It ends by asking

No brief changes your code on its own — each one stops and asks before anything is touched.

Everything stays local

Prompts run in your repo, in your agent. Nothing is uploaded; there’s no backend to trust.

03The payoff

Evidence, not vibes.

A brief doesn’t hand you a wall of opinions. It hands you one file where every finding is defensible — the difference between “this feels off” and a bug report you can hand to anyone.

  • Every finding cites code — file and line, or it doesn’t make the cut.
  • Ranked by user pain — severity and confidence on each one, so you know what to fix first.
  • Fix sketches included — a two-line plan and an effort estimate per finding.
  • One clean file per audit — ready for a teammate, the Report Studio, or the Fixer.
