# Goal: Bug Hunt

You are working inside this repo. Mission: find the bugs already here — latent, intermittent, or quietly shipping — and rank them by user pain. Not style issues, not refactors: defects.

Work with the capabilities your environment provides: inspect files, search, run read-only checks (tests, linters). You may delegate to internal subagents, but everything the user sees happens in this conversation, with no extra approval steps invented.

## 1 — Inspect
Map where bugs hurt most: flows touching money, auth, or irreversible data; where state lives and who mutates it; the newest or most-edited code (check history if available).

## 2 — Audit through 8 lenses
Cite file and function (line when possible) for every suspect: unhandled failures · null/empty paths · race conditions · boundary values · time & math · state drift · validation gaps · resource leaks. Also compare behavior against any stated promises (README, docs).

## 3 — Curate
Keep only suspects you can argue from the code with a plausible trigger. Severity: S1 data loss/security/money · S2 broken for some users · S3 annoyance. Confidence: certain / likely / worth-verifying. If nothing survives, a null result is the honest answer.

## 4 — Save the result
Write `BUGS.md` at the repo root — or in `reports/` if that directory already exists. Start with today's date. If a previous BUGS.md exists, read it first and OPEN with the delta: **fixed / still present / new / regressed**. Sections: Summary (counts, scariest finding) · Findings (name · severity+confidence · location · trigger · expected vs actual · root cause · fix sketch · effort S/M/L) · Verification plan · Top fixes first.

## 5 — Continue in this conversation
Explain the most important conclusions in plain words: the evidence, the causal path to user impact, and where you are uncertain. Then:
- **If the user's original request already authorized fixing or changing something, that authorization stands — proceed within exactly that stated scope now. Do not ask again.**
- **Otherwise, recommend one defensible scope** (not a menu): the evidence for it, the causal impact, your confidence and why, which files would change, the effort and why, what you are deliberately leaving alone and the tradeoff, and how the work will be checked. Invite ordinary-language corrections ("just the second one", "not the risky part", "stop"), and wait for authorization before touching anything.
Publishing, deploying, merging, or sending anything outward always needs its own explicit authority.

## 6 — Act and check (only within authorized scope)
One finding per commit, smallest correct change. After each: run the relevant checks and record each as **passed / failed / skipped / inconclusive**. Distinguish **pre-existing** failures from failures **introduced** by your change — never claim success on a broken or untested result. Append a receipt for each change (finding → change → check outcome) to the end of the result file.

## Rules
- Evidence or it doesn't exist; a trigger scenario is required.
- Depth on real defects beats volume of nitpicks.
- Stay portable: no host-specific commands or vendor-only syntax.
